## Technical Decisions

### Why ENABLE_REDIS flag?

**Option evaluated:** Check only REDIS_URL existence
**Problem:** REDIS_URL always set in CI/prod, can't opt-out without changing env vars

**Solution:** Explicit boolean flag
- Safer (default is OFF)
- Clear intent
- Can toggle without changing connection string
- Matches pattern of other feature flags

### Cache TTL Strategy (Current)

**Decision:** 24 hours for all resources
**Rationale:**
- Family members change rarely (weekly/monthly)
- Documents metadata stable (file URLs signed separately)
- Simplest to understand and troubleshoot

**Tradeoff:** Possible stale data for up to 24h
**Mitigation:** App works fine; manual refresh available

**Future improvement:** Smart TTLs by resource type
- Tasks: 1h (high churn)
- Family: 24h (rare changes)
- Subscriptions: 12h (medium churn)

### Invalidation Approach (Current)

**Pattern:** Manual invalidation on writes
- POST → invalidate
- PATCH → invalidate  
- DELETE → invalidate

**Benefit:** Simple, explicit, no hidden side effects
**Cost:** Must remember to invalidate in every mutation

**Future improvement:** Event-based invalidation
- Firebase Cloud Functions listen for changes
- Automatically invalidate Redis
- No manual invalidation needed

## Key Learnings

1. **Graceful degradation is powerful** - Redis is optional but when enabled, provides massive speed improvement with zero downtime risk

2. **Explicit feature flags > implicit behavior** - ENABLE_REDIS=true is clearer than "if REDIS_URL exists, enable caching"

3. **Manual invalidation is maintainable** - Event-based caching is more robust but requires more infrastructure; manual approach is sufficient for now

4. **Documentation matters for adoption** - Clear setup instructions and troubleshooting guide important for developer experience

5. **Cache hierarchy is critical** - App still works if Redis fails because we have Firestore fallback; this is more important than cache hit rates
