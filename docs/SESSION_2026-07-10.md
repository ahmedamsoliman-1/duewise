# Duewise development session — 2026-07-10

## Summary

Transformed Duewise from a passive record system into an active household workflow tool by implementing recurring workflows, batch actions, and proactive task completion patterns. The shared resource page now powers a cohesive multi-resource experience with consistent interaction patterns across tasks, subscriptions, documents, inventory, and more.

## Work completed

### 1. Recurring task and subscription workflows
**Objective:** Make household admin feel like a managed cycle, not chaotic deadlines.

**What was built:**
- Tasks and subscriptions now support recurrence intervals: weekly, monthly, yearly
- Recurrence end-date controls prevent infinite cycles
- When a task is marked "completed," the system automatically creates the next occurrence if recurrence is still active
- Same pattern for subscriptions: when marked "cancelled," the next occurrence is created
- Recurrence helpers in [lib/dates/recurring.ts](lib/dates/recurring.ts) with unit tests

**Key decisions:**
- Recurrence is a property on the task/subscription, not a separate collection (simpler, works with existing schema)
- Next occurrence is created server-side during PATCH request (consistent, single source of truth)
- Recurrence calculation includes date overflow handling (e.g., Jan 31 + 1 month = Feb 28, not March 3)

### 2. Shared resource page enhancements
**Objective:** Create a consistent interaction pattern across all collections.

**What was built:**
- All resource pages now share selection state, bulk actions, and duplicate workflows
- Multi-select checkboxes on grid cards, mobile cards, and table rows
- Persistent action bar showing selected count and available batch actions
- Drag-and-drop reordering in grid view with visual feedback
- Quick complete + batch complete for tasks

**Key decisions:**
- Selection state lives at the resource page level (not per-view)
- Drag-and-drop ordering uses visual position tracking, not a separate sort field
- Selection persists until explicitly cleared or after an action completes
- Same action names across all views ("Edit", "Delete", "Duplicate", "Complete") for consistency

### 3. Task completion workflows
**Objective:** Make task management feel proactive, not just record-keeping.

**What was built:**
- Single-click "Complete" button visible on all task views
- Batch completion for selected tasks
- "Complete and add next step" workflow:
  - Marks current task complete
  - Opens form for new task with pre-filled context:
    - Title prefixed with "Follow up:"
    - Notes carry forward from original task
    - Category, assigned family member, linked documents preserved
  - User can immediately edit and create the follow-up task
- Smart follow-up logic in [lib/tasks/follow-up.ts](lib/tasks/follow-up.ts)

**Key decisions:**
- Follow-up is a quick, low-friction workflow (not a hard link between tasks)
- Only available for tasks (not subscriptions, documents, etc.) initially
- Defaults are preserved from the original task but can be overridden
- Follow-up opens immediately after completion (no extra confirm step)

**Files changed:**
- [components/tables/resource-page.tsx](components/tables/resource-page.tsx) - task actions and workflows
- [lib/tasks/follow-up.ts](lib/tasks/follow-up.ts) - new helper for follow-up form prefill

### 4. Drag-and-drop ordering
**Objective:** Give users control over task and list prioritization.

**What was built:**
- Grid cards are draggable in desktop view
- Visual "Moving" state feedback during drag
- Drop zones on other items to reorder
- Ordering persists in the filtered list view
- Mobile and table views don't yet support drag-drop (can be added later)

**Key decisions:**
- Position-based ordering stored in a transient `position` field
- Drag-drop only in grid view (table/mobile too cluttered for this)
- No persistence to backend yet (can be added as a future enhancement)
- Simpler than row-swapping; more flexible for future grouping

## Architectural insights

### Resource page as a hub
The [components/tables/resource-page.tsx](components/tables/resource-page.tsx) component evolved into the central orchestrator for all resource workflows. It now manages:
- Form state and submission
- Multi-select state and batch actions
- Filtering, sorting, and view switching
- File uploads and storage integration
- Relation loading and rendering
- Drag-and-drop orchestration

**Lesson:** This component is approaching the practical size limit (~1,400 lines). Future refactoring should consider extracting:
- Batch action handlers into a custom hook
- Drag-drop logic into a separate utility
- Form state and submission into a custom hook

### Patterns that worked well
1. **Per-resource endpoints:** Each resource has its own `/api/{resource}/route.ts` for CRUD. Adding recurrence-aware logic here is clean.
2. **Shared schema validation:** Zod schemas in [lib/validators/schemas.ts](lib/validators/schemas.ts) are reused for form validation and API payloads.
3. **Consistent action callbacks:** `onEdit`, `onDelete`, `onDuplicate`, `onComplete` pattern makes it easy to add new batch actions.
4. **Position-based ordering:** Simpler than trying to manage explicit sort indices.

### Patterns that need refinement
1. **Mixed view logic:** ResourcePage handles grid, table, and mobile card rendering. A view context or hook might be cleaner.
2. **Inline form handling:** The edit form is inside the same component as the list. Extracting to a dialog or modal might improve readability.
3. **Relation caching:** Loading all relation options on mount might be inefficient at scale. Consider pagination or lazy loading later.
